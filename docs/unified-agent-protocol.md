# Unified Agent Protocol v1

This document defines the current internal protocol for AgentIsland as a product runtime.

The goal is to let Claude, Codex, Gemini, and future agents integrate through a provider translation layer without leaking provider-specific hook semantics into the core runtime.

Related docs:

- [Docs Index](./README.md)
- [Multi-Agent Architecture](./multi-agent-architecture.md)
- [Runtime Observability](./runtime-observability.md)

## Goal

AgentIsland should treat each external agent as a provider, not as a product mode.

The product core should consume:

- stable semantic events
- stable response actions
- explicit capability declarations

The product core should not depend on:

- raw official event names
- provider-specific permission wording
- provider-specific response JSON shapes

## Non-Goals

- perfectly erase all provider differences
- expose every provider field as a first-class core field
- pretend all providers support the same controls

Provider differences must remain visible through capabilities and `provider_payload`.

## Design Principles

- Normalize product semantics, not vendor terminology.
- Keep provider-specific parsing in ingress adapters.
- Keep provider-specific response encoding in egress adapters.
- Prefer capability-based behavior over `if provider == ...`.
- Support graceful degradation when a provider cannot honor a requested action.
- Preserve raw provider payloads for diagnostics and replay.

## Runtime Shape

Runtime flow:

`Provider Input -> Provider Ingress Adapter -> Unified Agent Event -> Core Engine -> Unified Agent Action -> Provider Response Adapter`

The core engine should only understand:

- unified events
- unified actions
- capability descriptors

## Protocol Layers

### 1. Provider Input

Raw provider-native data such as:

- Claude hook JSON
- Codex hook JSON
- Gemini hook JSON
- transcript metadata
- process metadata

### 2. Unified Agent Event

The stable event consumed by product logic.

### 3. Unified Agent Action

The stable response generated by product logic or policy logic.

### 4. Provider Response

Provider-native output emitted back to the originating runtime.

## Unified Event Envelope

Every normalized event should use one envelope.

```json
{
  "version": "1.0",
  "event_id": "evt_123",
  "timestamp": "2026-04-08T12:34:56Z",
  "provider": "codex",
  "session_id": "session_123",
  "turn_id": "turn_456",
  "kind": "permission.requested",
  "payload": {},
  "capability_hints": {},
  "provider_payload": {}
}
```

## Stable Envelope Fields

- `version`
  Protocol version.
- `event_id`
  Unique internal event identifier.
- `timestamp`
  Event creation time in ISO-8601.
- `provider`
  Source provider, for example `claude`, `codex`, or `gemini`.
- `session_id`
  Stable session or thread identifier.
- `turn_id`
  Optional turn identifier when the provider exposes one.
- `kind`
  Stable semantic event name.
- `payload`
  Stable semantic payload.
- `capability_hints`
  Optional event-scoped hints about what the provider can do for this event.
- `provider_payload`
  Raw provider-specific fields preserved for debugging, replay, and future migrations.

## Event Kinds

The core protocol should stay small and semantic.

### Session

- `session.started`
- `session.ended`
- `session.compaction_requested`
- `session.compacted`
- `session.cwd_changed`
- `session.config_changed`

### Turn

- `turn.input_submitted`
- `turn.started`
- `turn.completed`
- `turn.failed`

### Permission

- `permission.requested`
- `permission.resolved`
- `permission.denied_by_provider`

### Tool

- `tool.pending`
- `tool.started`
- `tool.completed`
- `tool.failed`

### Agent

- `agent.subtask_started`
- `agent.subtask_completed`
- `agent.idle`

### Model

- `model.request_prepared`
- `model.response_chunk`
- `model.response_completed`

### Notification

- `notification`

### Interaction

- `interaction.elicitation_requested`
- `interaction.elicitation_resolved`

These names are intentionally not tied to `PreToolUse`, `BeforeTool`, `PermissionRequest`, and similar provider names.

```json

```

This should only be emitted by providers that expose model-layer hooks.

## Unified Action Envelope

Every product-originated response should use one stable action object.

```json
{
  "version": "1.0",
  "action_id": "act_123",
  "target_event_id": "evt_123",
  "decision": "allow",
  "message": "Approved by AgentIsland",
  "continue": true,
  "stop_reason": null,
  "patch": {},
  "metadata": {}
}
```

## Stable Action Fields

- `version`
- `action_id`
- `target_event_id`
- `decision`
  One of `allow`, `deny`, `ask`, or `noop`.
- `message`
  User-facing or provider-facing explanation.
- `continue`
  Whether the turn or loop should continue when supported.
- `stop_reason`
  Optional reason when stopping.
- `patch`
  Structured modifications requested by the product.
- `metadata`
  Internal bookkeeping fields not meant as semantic controls.

## Patch Object

The `patch` field is where the protocol becomes more expressive than today's `decision/reason` pair.

```json
{
  "tool_arguments": {},
  "tool_result": {},
  "model_request": {},
  "model_response": {},
  "additional_context": "optional text",
  "tail_call": {
    "name": "tool_name",
    "args": {}
  }
}
```

A provider response adapter may:

- fully apply a patch
- partially apply a patch
- ignore unsupported patch fields

Unsupported fields must not crash the pipeline.

## Capability Descriptor

Every provider adapter must declare capabilities.

Recommended top-level shape:

```json
{
  "provider": "codex",
  "permissions": {},
  "tool_control": {},
  "model_control": {},
  "session_control": {},
  "agent_control": {},
  "history": {}
```

## Provider Translation Rules

Provider adapters should do two separate jobs.

### Ingress Adapter

Responsibilities:

1. Parse provider-native input
2. Determine semantic `kind`
3. Normalize stable payload fields
4. Attach capability hints when needed
5. Preserve raw provider fields in `provider_payload`

### Egress Adapter

Responsibilities:

1. Accept a unified action
2. Encode the closest provider-native response
3. Drop unsupported controls safely
4. Log downgrade decisions for diagnostics

## Downgrade Rules

The protocol must support intentional downgrade.

Examples:

- if the product requests `patch.tool_arguments` but the provider cannot rewrite args, ignore the patch and preserve the original execution path
- if the product requests `ask` but the provider only supports `allow` or `deny`, map to `noop`
- if the product requests `tail_call` but the provider cannot tail-call, discard it and record a diagnostic

Downgrade must be explicit in logs and observable in diagnostics.

## Codex Guidance

Codex should be modeled as a partially capable provider, not as a broken provider.

Current practical implications:

- `PreToolUse` is useful for runtime Bash approval
- app and tool approval policy also exists in configuration
- not every permission surface is emitted as a hook event
- hook response depth is narrower than Gemini and narrower than advanced Claude cases

Recommended modeling:

- map Bash approval requests to `permission.requested`
- expose config-derived policy as provider capability or provider policy state
- do not pretend AgentIsland controls all Codex permissions
- distinguish between:
  - runtime approval events
  - provider configuration policy
  - unsupported permission surfaces

This prevents product logic from assuming that Codex hook approval equals full Codex permission control.

## Claude Guidance

Claude exposes the richest hook surface today.

Recommended modeling:

- map rich event types into semantic kinds where product value exists
- keep long-tail event details in `provider_payload`
- avoid expanding the stable core for one Claude-only feature unless there is clear cross-provider value

Current implementation rule:

- Claude long-tail events do not enter the stable core by default
- they are promoted only when they start affecting shared product logic or show clear cross-provider reuse value
- until then, keep them in `provider_payload` and capability-level diagnostics

## Gemini Guidance

Gemini exposes strong model-layer and tool-layer transformation hooks.

Recommended modeling:

- use `model.*` events only when Gemini-like model hooks exist
- express argument rewriting, response rewriting, and tail calls through `patch`
- let providers without these capabilities degrade safely

## Migration Plan

### Phase 1

- keep the current hook bridge protocol intact
- define unified event and action types in parallel
- add provider capability descriptors

### Phase 2

- translate current `HookPayload` into unified events at the Swift boundary
- stop new UI logic from depending on raw provider fields

### Phase 3

- widen response handling beyond `decision/reason`
- add provider response adapters

### Phase 4

- move policy logic to capability-based decisions
- expose downgrade diagnostics

### Phase 5

- migrate bridge-side adapters to emit unified events directly if desired

## Rules for Stable Core Fields

Promote a field into the stable core only if:

- at least two providers can express it now, or likely soon
- it affects product logic, not just diagnostics
- the meaning survives provider translation without distortion

Otherwise keep it in:

- `provider_payload`
- provider-specific capability state
- diagnostics only

## Suggested Initial Swift Types

Suggested conceptual types:

- `UnifiedAgentEvent`
- `UnifiedAgentAction`
- `ProviderCapabilities`
- `ProviderIngressAdapter`
- `ProviderResponseAdapter`
- `ProviderPolicySnapshot`

These names are suggestions, not mandatory API names.

## Open Questions

- Should provider policy snapshots be event-driven or pulled on demand
- Should unified events include a separate `trace_id` for cross-event correlation
- Should tool risk classification happen in adapters, policy engine, or both
- Should unsupported action fields be reported inline to UI or only to diagnostics

## Summary

The product should standardize on:

- one semantic event protocol
- one semantic action protocol
- one explicit capability model

It should not standardize on:

- one fake provider behavior
- one fake permission system
- one fake response depth

That balance is what makes a multi-agent product architecture durable.
